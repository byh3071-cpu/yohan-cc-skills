# PAT-004 — 독트린·설정 SoT 단일화

## 핵심 한 줄
같은 규칙이 3곳에 흩어지면 반드시 모순·드리프트. **SoT 1곳 지정 → 파생엔 "SoT 여기" 표시 → 변경 시 전수 정렬**. 상충은 append 아니라 재작성으로 제거.

## 실사례 (모델 배분)
- 배분 독트린이 3곳 분산 + 모순: `CLAUDE.md`("계획·실행 모두 Opus, 품질 우선") vs `cost-guard`("구현/검증=sonnet") vs agent 핀(sonnet). critic을 opus로 올리려니 cost-guard와 정면 충돌.
- **처리**: 세 곳을 통합 tier 독트린("탐색=haiku, 구현=sonnet, 판단·검증=opus, 지휘=세션모델")으로 **정렬**(append 아님, 재작성).
- **파생 미러 전수 조사**(병렬 인벤토리): `docs/ARCHITECTURE.md`·`docs/PRD.md`가 배분 값 하드코딩 미러 → 동반 갱신 + "SoT=frontmatter" 명시. 로컬 다른 레포 자체사본 0, Notion 상충 1건(교차검증 프로토콜) 발견.
- **선례**: vhk `ADR-007`이 "yohan-core plugin이 SoT, vhk에 복제 금지 — SoT 2곳이면 드리프트"라고 이미 같은 원리를 명문화하고 있었음.

## 교훈 (역전파)
- 미러는 SoT 아님([[PAT-003]]과 자매) — 값 하드코딩 대신 "SoT 참조"로.
- 전역 상속(플러그인)일수록 정렬 필수: 모순도 전역 전파된다.
- 변경 전 **미러 전수 조사**(인벤토리)로 stale 후보를 빠짐없이 — 안 그러면 한 곳만 고치고 나머지가 어긋남.

## 비고
[[PAT-003]](상속 전제 실측 검증)의 자매. vhk ADR-007과 동일 원리(SoT 단일화)의 일반화. core 독트린 승격 후보. 카테고리 `state`는 근사치(거버넌스 전용 코드 부재). 노뚝이가 Notion 패턴 사전 등록.
