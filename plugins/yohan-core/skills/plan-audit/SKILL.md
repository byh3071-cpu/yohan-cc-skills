# plan-audit — 역추출 대조 감사

계획을 승인하기 전에 건다. **기존 `cross-check`와 시점이 다르다** — cross-check는 산출물의 내적 건전성, 이 스킬은 *요청 원문 ↔ 계획*의 정합.

## 왜 블라인드인가

잡으려는 결함은 계획의 버그가 아니라 **요청→계획 번역에서 떨어진 정보**다.
감사자가 계획을 먼저 읽으면 앵커링돼서 "빠진 것"은 구조적으로 못 본다.
그래서 **어느 서브에이전트에게도 계획을 주지 않는다.**

- `critic` → 사용자 발화만 받는다
- `explorer` → "이 파일 N행에 뭐가 있나" 같은 **사실 질의만** 받는다. 계획이 뭐라고 주장하는지는 모른다
- 대조는 지휘자(메인)가 한다

앵커링 차단은 인용 검증에도 그대로 적용된다. 지휘자가 직접 `sed -n '22p'`로 읽고 판정하면 계획을 이미 아는 상태라 "맞겠지" 하고 넘어간다. explorer는 계획을 모르니 사실만 돌려주고, 대조는 그 다음이다.

계획 사본 파일을 만들지 않는 게 핵심이다. 파일이 없으면 우연히 읽을 경로도 없다.

## 절차

### 0. 대상 확인
감사할 계획이 지휘자 컨텍스트에 있어야 한다. 없으면 **즉시 중단**하고 무엇을 감사할지 물어라.

### 1. 원문 추출
```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "<스크립트경로>"
```
스크립트 경로는 `$env:CLAUDE_PLUGIN_ROOT\skills\plan-audit\scripts\Export-UserTurns.ps1`.
**`${CLAUDE_PLUGIN_ROOT}` 리터럴을 그대로 넘기지 마라** — 확장되지 않아 죽는다.
변수가 비어 있으면 Glob으로 `**/skills/plan-audit/scripts/Export-UserTurns.ps1` 절대경로를 해석해 넘겨라.

산출물(`$env:TEMP\plan-audit\requests-<sid>.txt`)을 Read 한다. 스크립트가 수집하는 것:
- **직접 입력** (`promptSource=typed`)
- **반려 지시** — 사용자가 도구·계획을 반려하며 남긴 말. `tool_result`에 묻혀 있어 놓치기 쉽지만 **이 스킬의 핵심 입력**이다

실행 실패 시 폴백: 지휘자가 사용자 발화를 **verbatim**으로 직접 옮긴다. 요약하면 이 감사는 무의미해진다.

### 1b. 계획에서 검사 대상 뽑기
두 종류로 나눠 목록화한다. 성격이 달라 처리 방법이 다르다.

| 종류 | 예 | 처리 |
|---|---|---|
| **경로·심볼** (존재하나?) | `docs/patterns/PAT-001.md`, `Export-UserTurns.ps1` | 2-A. Bash |
| **인용** (그 내용이 맞나?) | `critic-gate.ps1:22`, "PAT-001이 X를 처방한다" | 2-B. explorer |

인용이 하나도 없으면 2-B를 스킵한다.

### 2-A. 존재 확인 — Bash 직접

LLM이 필요 없다. 판단이 안 들어가므로 서브에이전트로 돌리면 느리고 비싸기만 하다.
```bash
for p in <경로들>; do if [ -e "$p" ]; then echo "OK   $p"; else echo "MISS $p"; fi; done
```
심볼은 `Grep`으로 직접 확인한다.

### 2-B. 인용 검증 — explorer(haiku) 위임

**계획이 뭐라고 주장하는지 알려주지 마라.** 사실 질의만 던진다.
- ✅ "`plugins/yohan-core/hooks/critic-gate.ps1` 22행이 무슨 내용인지 그대로 인용해라"
- ✅ "`docs/patterns/PAT-001-*.md`의 '해결' 항목이 처방하는 게 뭔지 요약해라"
- ❌ "계획이 22행을 `permissionDecision='ask'`라고 인용했는데 맞나?" ← 답을 알려주면 앵커링

받은 사실을 지휘자가 계획의 주장과 대조한다.

### 2-C. 블라인드 요구사항 추출 — critic(opus) 위임

2-A·2-B와 **한 메시지에서 병렬**로 던진다. 서로 독립이다.

사용자 발화 전문을 프롬프트에 **verbatim 임베드**해서 넘긴다.
- 금지 지시를 반드시 포함: `~/.claude/plans/` 읽지 마라 · "계획/플랜" 파일 탐색하지 마라 · 무엇이 이미 설계됐는지 몰라야 한다
- 산출 요구: **요구사항 / 제약 / 성공조건 / 명시적 금지·비범위 / 모호한 지점** 5개 목록, 번호 부여
- 각 항목에 **원문 인용** 필수 — 근거 없는 항목을 원천 차단
- "발화에 근거 없으면 '발화에 없음'이라고 써라. 추측으로 채우지 마라"

원문이 50KB를 넘으면 임베드 대신 파일 경로를 넘긴다.

모델 티어는 **opus 유지**. 감사가 헐거우면 존재 이유가 없다. 요구사항 추출은 기계적으로 보이지만, 실측상 모호지점 11개를 잡아낸 건 opus급 판단이었다.

### 3. 대조
critic 목록 ↔ 계획을 매핑한다.

- 계획에 매핑 안 되는 요구사항 → **누락**
- 계획에 있는데 원문 근거 없음 → **스코프 크립** (단, 레포 관례·글로벌 규칙 유래면 그렇게 표기하고 통과)
- 요구사항이 다른 뜻으로 기술됨 → **왜곡**

> critic이 뽑은 항목만 *사용자 합의 사항*이다. 어시스턴트가 제안하고 사용자가 고른 것은 요구사항이 아니라 **채택된 제안**이다. 이 구분을 대조표에 남겨라.

### 4. 입도 검사
계획의 각 단계를 3기준으로 본다: **1커밋 · 독립 검증가능 · 롤백가능**.

### 5. 판정 출력

| # | 항목 | FAIL 조건 |
|---|---|---|
| 1 | 의도 정합 | 추출 요구사항 중 계획에 매핑 안 되는 항목 존재 |
| 2 | 스코프 크립 | 계획에 있는데 원문 근거도 출처 표기도 없는 항목 존재 |
| 3 | 왜곡 | 요구사항이 다른 뜻으로 구현되게 기술됨 |
| 4 | 분해 입도 | 1커밋·독립검증·롤백 중 하나라도 못 맞추는 단계 존재 |
| 5 | 근거 실존 | 부재 판정된 경로·심볼이 계획에 있음(2-A), 또는 인용 내용이 실제와 다름(2-B) |
| 6 | 미확인 가정 | 계획이 전제하는데 검증 안 된 사실이 명시 안 됨 |
| 7 | 검증 계획 | 통과 판정 기준 없는 단계 존재 |

**치명(1·3·5) 하나라도 걸리면 불통과.** `critic` 판정 규율과 같다.

모호한 지점이 남아 있으면 PASS라도 함께 표시한다 — 발화가 짧으면 기준선이 빈약해 거짓 PASS가 난다.

### 6. 2회차 재진입 — 여기서 멈추지 마라

5의 지적표를 근거로 **수정안을 작성한다. 계획에 적용하지는 않는다.**
그리고 **그 수정안을 대상으로 2~5를 한 번 더 돈다.** 사용자에게 물어보지 말고 바로 이어서 한다.

- 2회차 대상 = **1회차가 새로 쓴 문장뿐**. 손대지 않은 부분은 다시 안 판다
- 2회차에서도 결함이 나오면 수정안을 고치고, 그건 3회차 후보로 표시만 한다(자동 진행 금지)
- 2회차까지 끝나면 **두 회차 결함을 합쳐 한 번에** 사용자에게 낸다

**왜 적용하지 않고 제안에 머무는가:** 계획 수정은 사용자 승인 사항이다. 감사자가 계획을 고쳐 적용하면 승인 게이트를 우회하게 되고, 그건 이 스킬이 잡으려는 결함(스코프 크립)을 스스로 저지르는 꼴이다. 수정안 상태로 두면 2회차를 도는 데 아무 지장이 없다 — 실측상 2회차 결함은 전부 "1회차에서 새로 쓴 문장"에서 나왔고, 그 문장은 적용 여부와 무관하게 존재한다.

## 출력

한국어 반말·두괄식(`yohan-writing`). **지적표 → 수정안**까지. 계획 재작성·적용은 사용자 승인 후 지휘자가 한다 — 감사자가 저자가 되면 독립성이 무너진다.

1·2회차 결함을 회차별로 구분해 한 표에 낸다. 끝에 **남은 각도**(3회차 후보)를 제시하고 더 팔지는 사용자에게 넘긴다.

## 회차 운영 — 1·2회차는 호출 1번에 자동으로 돈다

한 번으로 안 끝난다. 실측상 회차마다 새 결함이 **직전 회차의 수정문**에서 나온다.
멈추는 기준은 "결함 0"이 아니라 **"새로 추가한 문장이 더 이상 새 결함을 안 낳을 때"**.

| 회차 | 대상 | 각도 | 자동/수동 |
|---|---|---|---|
| 1 | 원 계획 | 환각 스캔 + 블라인드 요구사항 대조 | **자동** — 호출 즉시 |
| 2 | **1회차 수정안**(적용 전) | 수정안이 새로 들인 인용·전제·근거 | **자동** — 절차 6으로 이어서 |
| 3+ | 누적 수정안 | 자기 루브릭 적용 · 실행 가능성 · 벤더 교차(Codex) | **수동** — 사용자가 결정 |

1·2회차는 각도가 고정이라 매번 똑같이 돌면 되므로 자동화한다. 사용자가 매번 "검증 및 리뷰"를 다시 치게 만들지 마라 — 실측상 그 반복이 이 스킬을 만든 이유다.

**1회차와 2회차는 대상이 다르다.** 1회차 2-B가 검사하는 건 *원 계획의* 인용이고, 2회차가 검사하는 건 *수정안이 새로 들인* 인용이다. "이미 인용 검증했다"고 2회차를 건너뛰지 마라 — 새 문장은 아직 아무도 안 봤다.

3회차부터는 각도 선택에 판단이 들어가므로 자동화하지 않는다.

## 경계

- 코드·diff 결함 역추적 → `cross-check` + `critic`
- 주장·의사결정 추론 검증 → `critical-thinking`
- 이 스킬은 **계획 승인 직전** 한 시점만 담당한다
